Why the Scheduler Matters

Both demos bounce a ball and fire 50,000 iterations of heavy CPU work. Watch how scheduleCallback keeps the animation silky-smooth by yielding to the browser, while the synchronous version freezes your frame budget entirely.

scheduleCallback
✅ With Scheduler
Heavy work is broken into small chunks and scheduled via scheduleCallback(NormalPriority, …). The scheduler yields every ~5 ms so rAF can fire and the animation never drops a frame.

import { scheduleCallback, NormalPriority } from "scheduler";

// Each chunk runs ≤ 5 ms then yields
function processChunk() {
  const deadline = performance.now() + 5;
  while (items.length && performance.now() < deadline) {
    heavyItem(items.pop());
  }
  if (items.length) scheduleCallback(
    NormalPriority, processChunk
  );
}

🧠 How the Scheduler Prevents Jank

⏱️

5 ms Time Slices

The scheduler checks performance.now() inside each task. When 5 ms has elapsed it stops, schedules the remainder, and returns control to the browser.

🎨

rAF Can Fire

Because the main thread is free between slices, requestAnimationFrame fires on schedule and the canvas ball animates without interruption.

🚦

Priority Lanes

ImmediatePriority, UserBlockingPriority, NormalPriority — the scheduler picks the most urgent task each slice, so user input is never starved.

📉

No Scheduler = Jank

A plain while loop runs to completion before the event loop can process paint, input or timers — you see a frozen UI.
