# Overview

This project explains how React's internal Scheduler works by separating the idea from the full React codebase. Scheduler is one of the pieces that helps React keep interfaces responsive while rendering work is in progress.

At a high level, Scheduler is a cooperative task manager. React registers units of work, assigns them priorities, and lets Scheduler decide when each task should run. Scheduler can pause between chunks of work, give the browser a chance to handle input or paint, and then continue the remaining work later.

Scheduler does not make JavaScript run in parallel, and it cannot interrupt a function that is already executing. Its power comes from organizing work into small callbacks, choosing the next callback by priority and expiration time, and yielding at controlled points.

## How This Project Is Organized

The project has two main parts:

- **Learn** explains Scheduler conceptually and then walks through the source code step by step. This is the best place to start.
- **API** documents the published package in this repository. After you understand the learning section, you can use the [API reference](/react-scheduler-explained/api/introduction) to explore the exported functions directly.

There is also a [showcase](/react-scheduler-explained/showcase) that demonstrates Scheduler behavior in practice and compares scheduled work with ordinary synchronous function execution.

## Source Code Used Here

The original React Scheduler source lives in the React repository: [facebook/react/packages/scheduler](https://github.com/facebook/react/tree/v19.2.4/packages/scheduler).

This project includes a focused copy of that implementation in the local [`src`](https://github.com/lukonik/react-scheduler-explained/tree/main/src) folder. The copy is based on React `19.2.4` and keeps the core scheduling behavior, including task queues, timer queues, priorities, host callbacks, and cooperative yielding.

Some testing and debugging details have been removed so the learning path can stay focused. The goal is not to hide how Scheduler works, but to make the essential control flow easier to study.

## Recommended Reading Path

Start with the Learn section. Read the prerequisite pages first if the event loop or min-heaps are unfamiliar, then move through the theory pages before opening the source-code walkthrough.

That order matters. Scheduler is easier to understand when you first know why React needs to yield, why tasks have expiration times, and why ready work and delayed work live in separate queues. Once those ideas are clear, the implementation becomes much easier to follow.
