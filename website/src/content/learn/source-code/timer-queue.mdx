# Timer Queue

`timerQueue` stores delayed tasks. These tasks are not ready to execute yet, so Scheduler keeps them separate from `taskQueue`.

```ts
var timerQueue: Array<Task> = [];
```

A delayed task enters `timerQueue` when `scheduleCallback` receives a positive `delay`:

```ts
newTask.sortIndex = startTime;
push(timerQueue, newTask);
```

Inside `timerQueue`, tasks are ordered by `startTime`. The earliest delayed task is always at the front of the heap.

## Why Delayed Tasks Do Not Execute Directly

When a delayed task reaches its `startTime`, Scheduler does not run it directly from `timerQueue`. Instead, it moves the task into `taskQueue`.

That matters because a delayed task may not be the most urgent ready task. Once it becomes eligible to run, it still needs to compete with other ready tasks by `expirationTime`.

The transition looks like this:

```ts
pop(timerQueue);
timer.sortIndex = timer.expirationTime;
push(taskQueue, timer);
```

The same task object is reused. Only its `sortIndex` changes:

- in `timerQueue`, `sortIndex` means `startTime`
- in `taskQueue`, `sortIndex` means `expirationTime`

## `advanceTimers`

The function that moves delayed tasks into `taskQueue` is `advanceTimers`:

```ts
function advanceTimers(currentTime: number) {
  let timer = peek(timerQueue);
  while (timer !== null) {
    if (timer.callback === null) {
      pop(timerQueue);
    } else if (timer.startTime <= currentTime) {
      pop(timerQueue);
      timer.sortIndex = timer.expirationTime;
      push(taskQueue, timer);
    } else {
      return;
    }
    timer = peek(timerQueue);
  }
}
```

It repeatedly checks the earliest delayed task.

If `timer.callback === null`, the task was cancelled, so Scheduler removes it from `timerQueue`.

If `timer.startTime <= currentTime`, the delay has elapsed. Scheduler removes the task from `timerQueue`, changes its `sortIndex` to `expirationTime`, and pushes it into `taskQueue`.

If neither condition is true, the earliest delayed task is still pending. Because `timerQueue` is ordered by `startTime`, every other delayed task starts even later, so `advanceTimers` can stop immediately.

## Where `advanceTimers` Runs

`advanceTimers` is called in multiple places:

- when a host timeout fires in `handleTimeout`
- at the beginning of `workLoop`
- after each completed task inside `workLoop`
- after a task returns a continuation

This keeps delayed tasks from getting stuck in `timerQueue` while Scheduler is already active.

## Summary

`timerQueue` is not an execution queue. It is a waiting area for delayed tasks. When a task's `startTime` arrives, `advanceTimers` promotes it into `taskQueue`, where normal priority and expiration ordering takes over.
